4.1 引言

目前为止，第1、2、3章分别讲述了计算机组成概念和性能计算公式、MIPS常见指令和算术操作。本章讲述处理器，其实现方式决定了时钟周期长度和CPI，我们将位MIPS指令集的两种不同实现方式分别建立数据通路和控制单元。

处理器本身的复杂意味着不可能剖析每个细节，故我们将从一个高度抽象和简化的概述开始，先建立数据通路，进一步构建简单的处理器实现MIPS这样的指令集，进而可以实现更复杂的指令集x86。

本章实际上涵盖面较广，我会选择性的学习。

一个基本的MIPS实现

首先实现MIPS指令集的一个核心子集：

  • 存储器访问指令：lw和sw。
  • 算术逻辑指令：add、sub、AND、OR和slt。
  • 分支指令：beq和j。

该子集可以说明建立在数据通路和控制单元时的关键原理，并可以在此基础上实现其他指令。

这些指令的实现过程大致相同，而与具体的指令类型无关，实现每条指令的前两步为：

  1. 程序计数器PC指向指令所在的存储单元，并从中去除指令。
  2. 通过指令字段内容，选择读取一个或两个寄存器。

对上述三种类型的指令来说，后续的步骤大致相同，这也体现了MIPS指令集的规整性。除跳转指令外，其他的指令都会通过ALU计算地址、执行运算或者进行比较，之后则根据ALU处理结果来读取或存储数据、改变下一条指令地址（默认为PC+4）。

4.2 逻辑设计的一般方法

在考虑计算机的设计时，必须决定及其的逻辑实现以及机器时钟。MIPS实现中的数据通路功能部件包括两种不同的逻辑单元：

  • 组合单元：一个操作单元，用于处理数据值，如与门或ALU。
  • 状态单元：一个存储单元，用于存储状态，如寄存器或存储器。

组合单元都用于处理数据值，其输出只取决于输入；状态单元则带有内部存储功能，关机后重启计算机状态单元中的值可回复，指令存储器、寄存器以及数据存储器等都是状态单元，一个状态单元至少有两个输入（待写入数据值和何时写入的时钟信号）和一个输出（已写入数据值）。

书中给出了时钟方法的概念，它规定了信号可以读出和写入的事件；进而给出了边沿触发的时钟方法的概念，用于简化后续描述。根据上述概念，可以将组合单元理解为状态单元的中间站，数据在组合单元中逗留、运算，却始终只在存储单元中保存。下图展示了组合逻辑（组合单元）、状态单元和时钟周期的关系（所有信号在一个时钟周期内从状态单元1到状态单元2）：

书中依次给出了执行R型指令、执行取数指令以及执行相等则分支指令时的过程。

R型指令(以 add $t1, $t2, $t3为例)

  1. 从指令存储器中取出指令，PC自增。
  2. 从寄存器堆中读出寄存器$t2和$t3。同时主控制单元计算出各控制信号的状态。
  3. ALU根据funct字段（指令的5:0位）确定ALU的功能，对从寄存器堆读出的数据进行操作。
  4. 将ALU的结果写入寄存器堆，根据指令的15:11位选择目标寄存器$t1。

取数指令（以 lw $t1, $offset($t2)为例）

  1. 从指令存储器中取指，PC自增。
  2. 从寄存器堆中读出寄存器$t2的值。
  3. ALU将从寄存器堆读出的值与符号扩展后的指令低16位值（offset）相加。
  4. 将ALU的结果作为数据存储器的地址。
  5. 存储单元的数据写入寄存器堆，目标寄存器由指令的20:16位（$t1）指出

相等则分支指令（以 beq $t1, $t2, offset）

  1. 从指令存储器中取指，PC自增
  2. 从寄存器堆读出寄存器$t1和$t2的值。
  3. ALU将从寄存器堆中读出的两数相减。PC+4的值与符号扩展并左移2位后的指令低16位（offset）相加，结果即分支目标地址。
  4. 根据ALU的零输出决定哪个加法器的结果存入PC中。

为什么不使用单周期实现方式

在单周期设计中，时钟周期对所有指令等长，故时钟周期长度由执行事件最长的指令确定，这意味着最低的效率。这条指令几乎可以肯定是取数指令，它依次使用了5个功能单元：指令存储器、寄存器堆、ALU、数据存储器、寄存器堆，虽然其CPI为1，但是其时钟周期很长，性能极差。

由于浮点数处理指令和其他更复杂指令的引入，大的指令集根本不可能使用单周期的实现方式。其性能瓶颈的本质在于指令集不能只满足部分指令，指令中最坏的情况也要兼顾。最坏的情况如取数指令在所有指令中占据了相当一部分，单周期实现方式违背了“加大大概率事件”的设计原则。

从而我们需要追求一种更聪明的实现方式——流水线。

4.5 流水线概述

流水线（pipelining）是一种实现多条指令重叠执行的技术。学完本节可以对流水线概念以及相关问题有大概的了解，第6到9节给出了相关细节。

书中以洗衣服的过程作为类比，展示了流水线技术对效率的提升，如下图所示：

观察可以发现，当任务数量相对于流水线级数不是很大时，流水线突然的启动和逐渐的结束会影响流水线的性能。将其扩展到MIPS指令的执行，一个MIPS指令通常包含以下步骤：

  1. 从指令存储器读取指令
  2. 指令译码的同时读取寄存器。MIPS的指令格式允许同时进行指令译码和读寄存器
  3. 执行操作或计算地址
  4. 从数据存储器读取操作数
  5. 将结果写回寄存器

面向流水线的指令集设计

下面是MIPS指令的四种特性：

  1. 所有的MIPS指令的长度相同。这简化了流水线的第一级取指和第二级译码。
  2. MIPS只有很少的几种指令格式，并且每一条指令的源寄存器字段位置都是相同的。这意味着在第二级确定取指类型的同时就可以读寄存器堆。
  3. MIPS中的存储器操作仅出现在存取指令中。相比于x86指令集，这简化了指令执行过程。
  4. 所有的操作数必须在存储器中对齐，因此不需要担心一个数据传输指令需要访问两次存储器的情况。（不太懂，mark）

流水线冒险

冒险（hazard）是指：在下一个时钟周期中下一条指令不能执行。下面将介绍三种流水线冒险：

  1. 结构冒险（structural hazard）：因缺乏硬件支持而导致指令不能在预定的时钟周期内执行的情况，即硬件不支持多条指令在同一时钟周期内执行。如在上述流水线中，若只有一个存储器（即数据和指令共用一个存储器），第4条指令的执行就会发生结构冒险，因为在一个存储器中同时进行了访存和取指。

  2. 数据冒险（data hazard）：因不能提供执行执行所需数据而导致指令不能再预定的时钟周期内执行的情况，发生在由于一条指令等待另一条指令的完成而造成流水线暂停的情况。如以下指令：

    加法指令直到第五步才写回其结果，这意味着浪费了3个时钟周期。一种解决方法为：一旦ALU生成了加法运算的结果，就可以将它作为减法运算的一个输入项。这称之为前推或者旁路。

对流水线概述的小结

流水线是一种在顺序指令流中利用指令间并行性的技术，与多处理器相比，其优势在于对程序员不可见。

下面6到9节将介绍流水线的一些概念、带来的问题、获得的性能提升，第10节将介绍一些高级流水线概念如超标量、动态调度等，第11节则将介绍一些最新的微处理器流水线。

在现代流水线中，结构冒险经常发生在浮点单元附近；控制冒险则一般发生在整数程序中，因为分支出现的频率更高；数据冒险在整数和浮点程序中都可能称为性能瓶颈，但在浮点程序中更容易处理。

书中补充了延迟（latency）的概念，即流水线的级数或者顺序执行过程中两条指令间的级数，表示单一指令的执行时间。正如前面所说，流水线的实现方式只能提高吞吐率而不能减少延迟。

正如前面所提到的，指令可以分为5个阶段：

  1. IF：取指令
  2. ID：指令译码
  3. EX：执行或计算地址
  4. MEM：数据存储器访问
  5. WB：写回

  • 写回阶段：将结果写回数据通路中间的寄存器中，这可能引发数据冒险。
  • 选择PC的下一个值时，需要在自增的PC和MEM级的分支地址间进行选择，这可能引发控制冒险

4.10 指令级并行

本书作者写的另一本教材《Computer Architecture: A Quantitative Approach》将本节的13页内容扩充到了200页。

流水线挖掘了指令间潜在的并行性，称之为指令级并行，有两种方法增加潜在的指令级并行程度：

  1. 增加流水线的深度（即级数）以堆叠更多的指令。如将周期较长的洗衣机级数分为洗、漂、甩三个部分，这样就变为了6级流水线。
  2. 复制计算机内部部件的数量，使得每个流水级可以启动多条指令，这被称之为多发射（multiple issue）。这使得CPI小于1，此时更好的度量方法是IPC。

实现多发射处理器有两种方式，其区别是将主要工作交给编译器来做还是硬件来做，由于两种实现方式导致某些决策分别为在编译时和在执行时进行，对应分别称为静态多发射和动态多发射。多发射处理器需要处理以下问题：

  1. 向发射槽发射多条指令：处理器如何确定在给定的时钟周期发射多少条指令以及发射何种指令。发射槽是指在给定时钟周期内能够发射指令的位置，可类比为短跑比赛的起点。
  2. 处理数据冒险和控制冒险。

推测的概念

推测是一种为了使依赖于被推测指令的其他指令可以执行，而允许编译器或处理器“猜测”指令结果的方法。推测意味着可能回猜错，故必须包含纠错机制，即能检查推测的正确性并在推测错误时能取消根据原推测结果执行指令的影响。

推测可由编译器或硬件完成，对应的错误恢复机制也有所不同，前者通过插入额外的指令检查推测的正确性并提供专门的修复例程，后者则通过缓存推测结果并对推测结果进行判断以确定是否需要重新执行。

推测也可能导致原本不存在的异常发生。书上没说清楚，mark

下面将介绍静态和动态的推测技术。

静态多发射处理器

静态多发射处理器都是用编译器来帮助封装多条指令并处理冒险，书中进而介绍了发射包（issue packet）和超长指令字（Very Long Instruction Word, VLIM）的概念。

编译器的任务包括静态分支预测和代码调度，以减少或阻止控制冒险和数据冒险。下面介绍一个简单的静态多发射MIPS处理器的例子。

动态多发射处理器

也称为超标量（superscalar）处理器。在最简单的标量处理器中，指令顺序发射，每个周期处理器决定是发射0条、1条还是多条指令。其仍然依赖处理器调度，但是是由硬件保证执行的正确性。

动态流水线调度（dynamic pipeline scheduling）是指“对指令进行重排序以避免阻塞的硬件支持”。

动态流水线调度

动态流水线调度选择下一条要执行的指令，可能的话会重排指令以避免阻塞。

流水线被划分为3个主要单元：取指与发射单元、多个功能单元和一个提交单元。其模型如下图所示：

在该流水线中使用了保留站和重排序缓冲区（提交单元的保留站）等概念，它们分别用于保存操作数与操作、暂时保存执行结果。下面是流水线中的两个关键步骤：

  1. 发射指令时，它先被复制到合适功能单元的保留站。如果其操作数在寄存器堆中或重排序缓冲区中，立即将其复制到保留站中。发射指令后，其操作数对应的寄存器可以被覆盖。
  2. 如果操作数不在寄存器堆和重排序缓冲区中，则应该被某个功能单元以计算结果的形式输出。硬件帮助定位该功能单元，在计算出结果后，直接将其复制到保留站，而跳过寄存器堆。

上述步骤可以有效利用重排序缓冲区和保留站实现前面静态流水线中提到的寄存器重名名。

在动态流水线中，处理器在不违背程序原有的数据流顺序的前提下以某种顺序执行各条指令，这种执行方式称为乱序执行，其执行指令的顺序与取指顺序不同。与乱序执行相对应的概念为顺序提交，它是一种较为保守的方案，也是所有的动态调度流水线所采取的方案。

动态调度经常与基于硬件的推测机制相结合，特别是对分支指令的推测。书上没说清楚mark。

能耗效率与高级流水线

4.14 谬误与陷阱

  • 谬误1：流水线是一种简单的结构

本章的诸多细节已经说明了设计流水线需要非常谨慎。前面提到的《Computer Architecture: A Quantitative Approach》对指令级并行和流水线有更为细致的描述。

  • 谬误2：流水线概念的实现与工艺无关

当今，随着晶体管规模的扩大，需要考虑功耗问题，因此不能使用太激进的设计。

  • 陷阱：没有考虑指令级的设计反过来会影响流水线

许多流水线中遇到的困难都是由指令级的复杂性造成的，例如：

  1. 指令长度和指令运行时间变化太大会导致各流水级的不均衡，从而阻碍某个流水级的运行，而且还会使冒险检测的实现复杂化。
  2. 复杂的寻址模式可能引起很多问题。更新寄存器的寻址模式会使冒险的检测复杂化，而需要多次访问存储器的寻址模式会使流水线的控制复杂化，并且难以保持流水线平稳流动。
  3. 两对实例：DEC Alpha和DEC NVAX，前者是后者性能的两倍；MIPS M/2000 和DEC VAX 8700，前者虽然执行更多的指令，但后者的平均时钟周期数为前者2.7倍，综合来看前者更快。

4.15 本章小结

处理器的数据通路和控制通路的设计，可以从指令集系统和对工艺基本特性的理解着手。从单周期实现到流水线，只是吞吐率的倍数提升，指令的执行时间并没有缩短；多发射增加了额外的允许每个时钟周期发射多条指令的数据通路硬件，但是却增加了有效延迟，其目的在于减少CPI。

流水线和多发射都旨在开发指令级并行，其限制因素均为存在数据相关和控制相关，破除限制的方式均为调度和推测。

书中展示了实例对指令级并行带来的性能提升作了严格描述。由于功耗墙的限制，使用更长的流水线、多发射和动态调度来维持处理器的性能增长力不从心，我们不得不在更粗粒度上开发并行性的多处理器（第6章）。根据Amdahl定律，处理器性能的不断增长势必使系统中的其他部件成为瓶颈——存储器（第5章）。
